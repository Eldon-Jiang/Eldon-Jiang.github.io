Big Ideas of Web Development

June 20, 2019 in Development

Website is just so interesting and important to the world nowadays, and I believe it is becoming more and more interesting and important every second.

So let’s talk about my understanding of web development; I want to explain the fundamental concepts in my way, Eldon’s way so that someone without experience would understand :p

Static or Dynamic

A Static website is one that is constant, showing exactly the same contents and performing exactly the same tasks to every one. My blog is a perfect example of a static website – it’s the same website unless I change the contents of it. Since the website shows the same contents, it normally does not require a server to retrieve other resources and already contain all information it needs when it is being deployed.

A Dynamic website is one that changes, showing different contents and performing different tasks for each different session. Facebook is a perfect example of a dynamic website – no developer is changing the web page for you every day (well hopefully…), the website has the same design but the actual contents are being retrieved from a server.

The key difference between them is that the Static website already knows the required information for it to render when it is deployed but the Dynamic website does not know until it retrieves the information from the server.

Frontend, Backend and Full-stack

Most web applications are dynamic because each user has their own user data. For this purpose to be fulfilled, a server that can store data and communicate with the ‘website’ should exist; and from here we can divide our web app development into the so called Frontend and Backend.

The ‘genius’ developers who can work with both and build an entire web application on their own are the so called Full-Stack Developers (just kidding, no developer truly work on their own :P). To put it simply, the user should be see and fell the Frontend work but never realize the Backend exist.

Imagine you are in a fancy restaurant, Frontend would be the nice-looking waiter in front of you who give the menu to you, serve you, take your order and deliver dishes to you (well also charge you money); Backend would be the chef in the kitchen who look at your order given by the waiter, cook the food and give the food to the waiter. The waiter and the chef work together so that you can enjoy the delicious dishes.

For the Frontend (or the waiter), giving you the menu is to render a User Interface, serving you is to emit events responding to your actions, taking your order / delivering dishes to or from the kitchen is to send API request and update UI based on API response.

For the Backend (or the chef), lookin at the order is to listen to API request to respond, cooking the food based on recipe is to perform tasks logically and manipulate the data, and giving the food back is to return the API call’s response.

Working together, the Frontend and the Backend serve the client a fluent and seamless user experience and make countless great things possible.

A really important part of the backend is the Database which is a like a place to store data, and it is manipulated by the backend application. The backend application and database are running on a server thus no user can directly access them without visiting the frontend website.

HTML, CSS and JavaScript

HTML, CSS and JavaScript are the fundamental building blocks of any web application, more precisely for the Frontend. They are received by the browser when it connects to a website; and a browser can use them to render the view seen by the users.

No matter what fancy framework is used to build the web app frontend, it eventually need to give the browser html, css and JavaScript files; which tell the browser about webpage what contents it has, how the contents look like and how to respond to the user’s actions respectively.

RESTful API

RESTful API, application program interface using HTTP request, is used widely for frontend and backend to communicate. Usually they are divided to different Http Action; GET, PUT, POST, DELETE are the most common kinds.

These APIs are usually used to manipulate database, and are very much self-explanatory: GET is to retrieve data; POST is to add data in; PUT is to update data; DELETE is to delete data.

The API request is sent by the frontend to a specific url with the necessary query headers and request body when certain events trigger it. The backend application is running on a server and listening to all the incoming API requests; then it would act accordingly to perform its tasks and then send a API response back to the frontend.

And sometimes there is a API Gateway on the server that looks at the API first and takes care of the the routing and authentication; in simpler words, the Gateway service decides which application should take care of this request and if the request has the correct authorization to be performed.

And here we go, I have talked about some of the most important concepts about web development. For more detailed and technical description about each part of web development, I want to cover them in other posts! Don’t hesitate to discuss anything with me!
